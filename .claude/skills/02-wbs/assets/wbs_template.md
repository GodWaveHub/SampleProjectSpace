## 1. WBS作成方針

### 1.1 目的
- ユースケース/成果物起点で、実行可能かつ測定可能なタスク分解を行う
- 進捗・工数・リスクを可視化し、プロジェクト管理を効率化する

### 1.2 作成基準
- **粒度**: 1タスクあたり1～3人日（最大5人日）、8時間未満は原則まとめる
- **階層**: 2～3層を推奨（ParentIDで管理）
- **命名**: 動詞で開始し、成果物を明示（例: 「○○仕様書を作成」）
- **依存関係**: IDで明示し、クリティカルパスを特定
- **完了条件**: 明確なDoD（Definition of Done）を記載

### 1.3 参照ドキュメント
- `PARAMETER.md`: 工数・工程配分・バッファ率を参照
- `01_Estimate/`: 見積結果を基にタスク化
- `.github/instructions/wbs_instructions.md`: 詳細ルール

## 3. タスク種別の定義

| 種別 | 説明 | 例 |
|------|------|-----|
| 管理 | プロジェクト管理活動 | 計画策定、進捗管理、課題管理 |
| 要件 | 要件定義活動 | ヒアリング、要件整理、要件定義書作成 |
| 設計 | 設計活動 | 基本設計、詳細設計、DB設計 |
| 実装 | コーディング活動 | プログラミング、単体テスト |
| テスト | テスト活動 | 結合テスト、システムテスト |
| 運用 | 運用・保守活動 | 受入支援、本番移行、保守 |

**参照**: IPA「共通フレーム2013（SLCP-JCF2013）」

## 4. 状態の定義

| 状態 | 説明 |
|------|------|
| TODO | 未着手 |
| DOING | 作業中 |
| DONE | 完了 |
| BLOCKED | ブロック中（依存タスク未完了、課題未解決等） |

---

## 5. 優先度の定義

| 優先度 | 説明 |
|--------|------|
| H（High） | 高優先度（クリティカルパス、必須タスク） |
| M（Medium） | 中優先度（通常タスク） |
| L（Low） | 低優先度（後回し可能） |

## 6. 工数・期間の計算方法

### 6.1 工数算出

**基本工数（人日）** = (LOC ÷ 生産性) × 補正係数

**参照**: `PARAMETER.md` 3.4節

### 6.2 期間算出

**期間（日）** = 工数（人日） ÷ 並行作業可能人数

**例**:
- 工数20人日、2名で作業 → 期間10日
- 工数10人日、1名で作業 → 期間10日

### 6.3 バッファの扱い

- スケジュールバッファ: 15～20%（`PARAMETER.md` 5.1節参照）
- 管理タスクとして別途計上（「予備タスク」「バッファ」等）

## 7. 依存関係とクリティカルパス

### 7.1 依存関係の種類

| 種別 | 略称 | 説明 |
|------|------|------|
| Finish to Start | FS | 前タスク完了後に開始（標準） |
| Start to Start | SS | 前タスク開始と同時に開始 |
| Finish to Finish | FF | 前タスク完了と同時に完了 |

**原則**: FSを使用。SS/FFが必要な場合は備考に記載。

### 7.2 クリティカルパスの特定

- 依存関係を追跡し、最長経路を特定
- クリティカルパス上のタスクは優先度Hに設定
- 遅延時の影響を備考に記載

---

## 8. 完了条件（DoD: Definition of Done）

各タスクには明確な完了条件を記載してください。

**例**:
- 「レビュー承認」: レビュー会議で承認され、議事録作成完了
- 「全ケース合格」: テスト実施結果が全て合格（Pass率100%）
- 「ドキュメント承認」: ステークホルダーの署名・承認を取得

---

## 9. 成果物の管理

### 9.1 成果物の保存先

成果物は以下のフォルダに保存してください：

| 成果物種別 | 保存先 |
|-----------|--------|
| 見積書 | `01_Estimate/` |
| WBS | `02_WBS/` |
| スケジュール | `03_Schedule/` |
| 要件定義書 | `01_Estimate/` または `07_Design/` |
| 基本設計書 | `07_Design/01_BasicDesign/` |
| 詳細設計書 | `07_Design/02_DetailDesign/` |
| DB設計書 | `07_Design/03_DatabaseDesign/` |
| ソースコード | `08_Project/` |
| テスト仕様書・結果 | `09_Test/` |

### 9.2 成果物のバージョン管理

- Gitでバージョン管理
- 重要な成果物（設計書等）はバージョン番号を付与（v1.0, v1.1等）

## 10. リスク管理

### 10.1 リスクの記載

各タスクのリスク列に主要リスクを記載してください。

**例**:
- 「要件変更」: 顧客からの追加要求による工数増
- 「技術課題」: 未経験技術による実装遅延
- 「バグ多発」: 品質低下によるテスト期間延長

### 10.2 リスク対応

- 高リスクタスクは早期着手
- リスク顕在化時の対応策を備考に記載
- リスク管理表は`04_Issues/`で管理

---

## 11. 前提条件・制約事項

### 11.1 前提条件

各タスクの前提列に、実行に必要な前提を記載してください。

**例**:
- 「見積完了」: 見積書が承認されていること
- 「要件確定」: 要件定義が確定していること
- 「顧客対応可能」: 顧客担当者がアサインされていること

### 11.2 制約事項

各タスクの制約列に、制約条件を記載してください。

**例**:
- 「開発環境は顧客提供」
- 「土日作業不可」
- 「特定メンバー専任」

---

## 12. 更新・レビュー

### 12.1 更新ルール

- WBS変更時はバージョンアップ（v1.0 → v1.1）
- 変更履歴に変更内容を記載
- 状態・日付・工数を最新化

### 12.2 定例レビュー

- **週次**: 進捗確認、BLOCKED解除
- **月次**: 工数・スケジュール実績との差異分析
- **マイルストーン**: 完了条件達成確認
